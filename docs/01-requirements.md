# 要件定義書 — 飲食店マーケティングOS

## 1. システム概要

### 1.1 目的
「飲食店マーケティングOS」は、飲食店の売上向上・集客改善を支援するマーケティング会社が、自社のコンサルティング業務を一元管理するための社内業務システムである。見込み顧客の獲得から、契約、ヒアリング、店舗診断、戦略設計、施策実行、数値管理、月次レポートまでの一連のコンサルティングプロセスを標準化・可視化し、担当者間のナレッジ共有と業務品質の平準化を実現する。

### 1.2 解決したい課題
- コンサルティングのプロセスが担当者ごとに属人化しており、品質にばらつきがある
- クライアント・店舗ごとの情報（ヒアリング内容、診断結果、施策、数値）が分散して管理されている
- KPIの進捗や施策の効果が横断的に把握できず、注意が必要な店舗を早期に発見しにくい
- 月次レポートの作成に工数がかかっている
- 成功事例・失敗事例のナレッジが蓄積・再利用されていない

### 1.3 提供価値
- コンサルティングプロセスの標準化（見込み顧客 → 契約 → 診断 → 戦略 → 施策 → レポート）
- 複数クライアント・複数店舗の横断管理と、注意店舗の早期発見
- 数値（KPI）の時系列管理と施策効果の可視化
- 月次レポート作成の効率化
- ナレッジの蓄積と再利用

## 2. 対象ユーザーと権限

### 2.1 想定ユーザー
| ロール | 日本語名 | 説明 |
| --- | --- | --- |
| admin | 管理者 | マーケティング会社の管理者。全機能・全データにアクセスでき、ユーザー管理・組織設定を行う。 |
| marketer | マーケティング担当者 | 実際にコンサルティングを担当するメンバー。担当クライアント・店舗の情報を作成・編集する。 |
| client | クライアント（将来） | クライアント企業の担当者。自社に関わるデータの一部を閲覧する。Phase 3 で提供予定。 |

### 2.2 権限方針（RBAC）
- ロールベースのアクセス制御（RBAC）を採用する。
- すべてのデータは `organization_id` によるテナント分離を前提とし、他組織のデータには一切アクセスできない。
- `admin` は自組織の全データにアクセス可能。ユーザー招待・ロール変更・組織設定が可能。
- `marketer` は自組織のクライアント・店舗データを閲覧・編集可能。ユーザー管理・組織設定は不可。
- `client` は自社（紐づく client）に関するデータのうち、共有が許可された範囲のみ閲覧可能（Phase 3）。

## 3. 機能要件

### 3.1 見込み顧客管理（Lead Management）
- 見込み顧客（リード）の登録・一覧・詳細・編集・論理削除
- ステータス管理（新規 / 商談中 / 提案済 / 受注 / 失注）
- 担当者の割り当て、次回アクション・メモの記録

### 3.2 商談・契約管理（Contract Management）
- 商談の進捗管理、契約情報（契約期間、金額、プラン、更新日）の登録
- 見込み顧客からクライアントへの転換（コンバージョン）
- 契約更新・解約の記録

### 3.3 初回ヒアリング（Hearing）
- クライアント・店舗ごとの初回ヒアリングシートの作成・保存
- 事業概要、ターゲット、現状の集客手段、課題認識、目標、予算などの項目管理
- テンプレートに基づく構造化入力

### 3.4 店舗診断（Store Diagnosis）
- 店舗の現状を体系的に評価する診断機能
- 5つの価値カテゴリ（後述）に基づく診断項目のスコアリング
- 診断結果の可視化（レーダーチャート等）と総評コメント

### 3.5 市場・商圏・競合分析（Market / Trade Area / Competitor Analysis）
- 市場分析：市場規模、トレンド、成長性の記録
- 商圏分析：商圏人口、世帯構成、動線、昼夜間人口などの記録
- 競合分析：競合店舗のリストアップ、強み・弱み、価格帯、ポジショニング比較

### 3.6 5つの価値分析（Five Values Analysis）
- 店舗の提供価値を「5つの価値カテゴリ」の観点で分析する。
- 各カテゴリに紐づく診断項目をスコアリングし、強み・弱みを構造的に把握する。
- カテゴリ例（組織ごとにカスタマイズ可能）：
  1. 商品・メニュー価値
  2. 接客・サービス価値
  3. 空間・雰囲気価値
  4. ブランド・情報発信価値
  5. 価格・コストパフォーマンス価値

### 3.7 課題特定（Issue Identification）
- 診断・分析結果から課題を抽出・登録
- 課題の優先度、影響度、緊急度、カテゴリ分類
- 課題と戦略・施策の紐づけ

### 3.8 マーケティング戦略設計（Strategy Design）
- 課題を踏まえた戦略の立案・登録
- 戦略の目的、ターゲット、コンセプト、期待効果の記述
- 戦略と課題・KPI・施策の紐づけ

### 3.9 KPI設定（KPI Management）
- KPI定義（指標名、目標値、単位、集計方法、対象期間）の登録
- KPIの月次実績値の入力と時系列管理
- 目標対比の進捗可視化

### 3.10 施策実行管理（Initiative Management）
- 施策の登録（目的、内容、期間、担当、予算、ステータス）
- 施策とKPI・戦略・課題の紐づけ
- 施策の進捗・実績の記録

### 3.11 SNS / MEO / 広告 / LINE 施策
- チャネル別施策（SNS投稿、MEO対策、Web広告、LINE配信）の管理
- SNSコンテンツの企画・下書き・投稿予定・投稿済みの管理
- チャネルごとの数値（フォロワー、リーチ、クリック、来店等）記録

### 3.12 数値管理（Metrics Management）
- 店舗ごとの主要数値（売上、客数、客単価、来店数、予約数等）の月次記録
- KPIとの連携、時系列グラフ表示

### 3.13 月次レポート（Monthly Report）
- 月次レポートの作成・編集・承認・共有
- KPI実績、施策実行結果、次月アクションの自動下書き生成（将来はAI活用）
- PDF出力、クライアントへの共有

### 3.14 ナレッジ蓄積（Knowledge Base）
- 成功事例・失敗事例のナレッジ登録
- 業種・課題・施策タグによる検索・再利用
- 診断・戦略設計時の参考情報として活用

### 3.15 複数店舗・複数クライアント横断管理
- ダッシュボードでの横断可視化
- 注意が必要な店舗（KPI未達、期限超過タスク等）の抽出
- 担当者別・クライアント別の進捗一覧

## 4. 非機能要件

### 4.1 セキュリティ
| 項目 | 要件 |
| --- | --- |
| 認証必須 | Supabase Auth による認証を必須とする。未認証ユーザーはログイン画面へリダイレクトする。 |
| RBAC | ロール（admin / marketer / client）に基づくアクセス制御を行う。 |
| テナント分離 | すべてのデータは `organization_id` により論理的に分離する。API/クエリで常に `organization_id` を条件に含める。 |
| Supabase RLS | Row Level Security を全テーブルで有効化し、DBレイヤーで `organization_id` とロールによるアクセス制御を強制する。 |
| バリデーション | 入力値は Zod スキーマでサーバー・クライアント双方で検証する。 |
| 操作ログ | 重要な作成・更新・削除操作は `activity_logs` に記録する（誰が・いつ・何を）。 |
| ファイル制限 | アップロードファイルは形式（画像・PDF等）とサイズ上限を制限する。Supabase Storage を利用。 |
| 論理削除 | 主要エンティティは物理削除せず `deleted_at` による論理削除とする。 |

### 4.2 性能・可用性
- 一覧画面はページネーション／サーバーサイド絞り込みで大量データに対応する。
- ダッシュボードの集計は必要に応じてキャッシュ・集計テーブルを検討する。

### 4.3 保守性・拡張性
- 型を厳密に定義し `any` を禁止する。
- コンポーネントを適切に分割し、UIキット（`components/ui`）を再利用する。
- 定数は `constants` に集約し、マジックナンバー・文字列を排除する。
- 診断項目・KPI・価値カテゴリは組織ごとにカスタマイズ可能な設計とする。

### 4.4 UX
- すべてのデータ取得画面でローディング・エラー・空状態を用意する。
- 大量項目を持つフォーム（ヒアリング・診断）はセクション分割・自動保存・進捗表示を検討する。

### 4.5 国際化・その他
- UIは日本語を第一言語とする。
- 日付・数値・通貨は日本ロケールで表示する。
