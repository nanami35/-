# 今後の開発ロードマップ — 飲食店マーケティングOS

本ドキュメントは、フェーズごとの開発ステップと、モックデータ層から本番（Supabase/Prisma）への移行、AI統合の方針を定義する。

---

## Phase 0：基盤構築（開発準備）

**ゴール**: 開発の土台を整え、モックで全画面が動く状態にする。

1. Next.js（App Router）+ TypeScript + Tailwind + shadcn/ui 風UIキットの初期化。
2. ルーティング骨格（`(auth)` / `(app)` レイアウト、サイドバー、ヘッダー）の構築。
3. `src/types`（ドメインモデル型）と `src/lib/constants.ts`（メニュー・ステータス・ロール）の定義。
4. `src/lib/sample-data/*` のモックデータ整備と、`src/lib/data/*`（DAL）のモック実装。
5. 共通UI（ローディング・エラー・空状態・テーブル・フォーム）の用意。

**完了条件**: モックデータで全画面が閲覧でき、ナビゲーションが機能する。

---

## Phase 1：MVP（中核フローの実装）

**ゴール**: クライアント → 店舗 → ヒアリング → 診断 → 課題 → KPI → 施策 → タスク → 月次レポートの一連を、モック層上で操作可能にする。

1. 認証（Supabase Auth）とロール判定・ルートガードの実装。
2. ダッシュボード（注意店舗・期限超過タスク・KPI進捗）。
3. クライアント管理・店舗管理（一覧・詳細・作成・編集・論理削除）。
4. ヒアリング（テンプレート入力・保存）。
5. 店舗診断（5つの価値スコアリング + レーダーチャート）。
6. 課題管理・KPI管理（定義 + 月次実績）・施策管理・タスク管理。
7. 月次レポート（下書き・編集・承認・PDF出力）。
8. RBAC（admin / marketer）による画面・操作制御。

**完了条件**: 「MVPのゴール」（02-mvp-scope.md）を満たす。

---

## Phase 1.5：モックデータ層 → 本番接続（Supabase / Prisma）

**ゴール**: DALの実装をモックから本番DBへ差し替え、実データで動かす。

1. `prisma/schema.prisma` を 05-er-diagram.md に沿って実装し、マイグレーションを作成。
2. `prisma/seed.ts` でデモ組織・ユーザー・サンプルデータを投入。
3. `src/lib/data/*` の各関数を Prisma / Supabase 実装に切替（環境フラグ `USE_MOCK` で分岐）。
4. **Supabase RLS の有効化**:
   - 全テーブルで RLS を ON。
   - `organization_id` = ユーザー所属組織のみ許可するポリシーを定義。
   - ロール（admin/marketer/client）による read/write ポリシーを追加。
   - アプリ層でも `organizationId` を全クエリに付与し二重防御。
5. `activity_logs` への操作記録、`deleted_at` による論理削除の実装。
6. Supabase Storage を用いたファイルアップロード（形式・サイズ制限）。

**完了条件**: 実DB上でMVP機能が動作し、テナント分離がRLSで保証される。

## Phase 3：AI統合・自動化・外部連携

**ゴール**: 業務を高度化・自動化する。

### 3.1 AI統合（Human-in-the-loop 必須）
対象：診断コメント生成、課題抽出案、戦略案、月次レポート下書き。

- **モデル**: OpenAI（GPT系）または Anthropic Claude を利用。用途に応じて選択・切替可能な抽象化レイヤ（`src/lib/ai/*`）を設ける。
- **設計原則（人間の承認を必須とする）**:
  1. AIは常に「下書き（draft）」を生成するのみで、確定・保存・共有はしない。
  2. 生成物は必ず担当者が編集できるUIに表示する（編集前提）。
  3. 保存・確定は人間の明示操作（承認ボタン）を経て初めて反映される。
  4. AI生成物であることをメタ情報として保持し、`activity_logs` に生成・承認の記録を残す。
  5. 入力に含める顧客データは最小限とし、テナント境界を越えたデータをプロンプトに混在させない。
- **フロー例（月次レポート）**:
  ```
  KPI実績・施策結果を集約
    → AIが考察・次月アクションの下書き生成（draft）
      → 担当者が編集
        → 管理者が承認（approved）
          → PDF出力・共有
  ```

### 3.2 外部連携
- Google Business Profile（MEO）、SNS各種API、Web広告媒体、予約/POSデータ連携。
- 数値の自動取り込みによりKPI入力工数を削減。

### 3.3 クライアント専用画面・通知・自動送信
- `client` ロール向け閲覧ポータル（診断・KPI・レポート）。
- KPI未達・期限超過・承認依頼のアプリ内／メール通知。
- 承認済み月次レポートのクライアントへの自動送付。
